<objective>
Add four symbol-aware code intelligence MCP tools -- `localnest_find_callers` (SYM-01), `localnest_find_definition` (SYM-02), `localnest_find_implementations` (SYM-03), `localnest_rename_preview` (SYM-04) -- powered by a new `symbol_index` SQLite table populated during project indexing. Symbol extraction uses a regex-first approach (proven patterns from existing `symbol-search.ts`) with optional tree-sitter AST enhancement for higher precision when grammar packages are available (SYM-05). All queries resolve to parameterized SQLite SELECTs against the indexed table, meeting the 500ms performance target on 1000-file repos (SYM-06).

Purpose: Give agents precise, IDE-grade code navigation without leaving the MCP session -- find where a function is called, jump to its definition, discover all implementations of an interface, and preview rename impacts before touching files. This is the highest-risk phase in v0.2.0 due to the tree-sitter dependency, mitigated by the regex-first design ensuring all tools work even without tree-sitter.

      - extractSymbolsTreeSitter(filePath, text, languageId, parser): walks the AST using collectCandidateNodes pattern. For each declaration node, extracts name via getNodeName, scope via buildScopePath, start/end lines. Also walks for call_expression / function_call nodes to find call sites.       - extractSymbolsFromFile(filePath, text, astChunker?): orchestrator function. Resolves languageId from extension. Tries tree-sitter extraction if astChunker provided. Falls back to regex extraction. Merges results: tree-sitter entries win over regex when both cover the same (file, symbol, line).

    Regex extraction strategy per language:
    - TypeScript/JavaScript: scan for function/class/interface/type/enum/const declarations + export statements. Call sites via \bname\s*\(.
    - Python: scan for def/class. Call sites via \bname\s*\(.
    - Go: scan for func/type declarations. Call sites via \bname\s*\(.
    - Rust: scan for fn/struct/enum/trait/impl declarations. Call sites via \bname\s*\( or name::method\s*\(.

    The regex extractor processes the file line-by-line. For each definition found, it records:
    - symbol_name: the identifier name
    - symbol_kind: mapped from the keyword (function->function, class->class, interface->interface, etc.)
    - is_definition: 1
    - is_export: 1 if line contains export/pub/Public keyword
    - line_start/line_end: the line number (single line for regex)
    - context_text: the trimmed line text, capped at 200 chars
    - scope_path: empty for regex (only tree-sitter can produce this)

    For call sites found during a second pass:
    - symbol_kind: 'call'
    - is_definition: 0
    - context_text: the trimmed line

       ```ts
       export const SYMBOL_INDEX_SCHEMA = `
         CREATE TABLE IF NOT EXISTS symbol_index (
           id TEXT PRIMARY KEY,
           file_path TEXT NOT NULL,
           symbol_name TEXT NOT NULL,
           symbol_kind TEXT NOT NULL,
           node_type TEXT,
           line_start INTEGER NOT NULL,
           line_end INTEGER NOT NULL,
           scope_path TEXT DEFAULT '',
           language TEXT NOT NULL,
           is_definition INTEGER DEFAULT 0,
           is_export INTEGER DEFAULT 0,
           context_text TEXT DEFAULT '',
           indexed_at TEXT NOT NULL,
           UNIQUE(file_path, symbol_name, line_start, symbol_kind)
         );
         CREATE INDEX IF NOT EXISTS idx_symbol_name ON symbol_index(symbol_name);
         CREATE INDEX IF NOT EXISTS idx_symbol_file ON symbol_index(file_path);
         CREATE INDEX IF NOT EXISTS idx_symbol_kind ON symbol_index(symbol_kind);
         CREATE INDEX IF NOT EXISTS idx_symbol_lang ON symbol_index(language);
       `;

       export function ensureSymbolIndexTable(db: { exec(sql: string): void }): void {
         for (const stmt of SYMBOL_INDEX_SCHEMA.split(';').filter(s => s.trim())) {
           db.exec(stmt.trim() + ';');
         }
       }
       ```
