<objective>
Add a transactional batch write tool for memory entries (BATCH-03) that accepts up to 100 memories per call, runs the entire batch inside a single adapter.transaction (BATCH-04), reuses the existing fingerprint + semantic dedup pipeline so duplicate items increment a counter instead of failing the batch, and preserves the recently-shipped nest/branch pass-through behavior (quick task 260409-pdf regression guard).

Purpose: Memories are the heaviest payload in v0.2.0 — agents capturing 50-100 memories at the end of a session must not pay per-row MCP serialization overhead. This unblocks Phase 27 (terse responses) and Phase 29 (KG fusion auto-linking) which both depend on bulk memory ingestion.

Output: A new `entries-batch.ts` module under 500 lines, store/service plumbing, one new MCP tool (`localnest_memory_store_batch`), and a test file covering happy path, fingerprint dedup, semantic dedup, validation errors, transaction rollback, oversized batch, and the nest/branch pass-through regression.
</objective>

    - storeEntryBatch({ memories, response_format? }):
      - Rejects when memories.length > 100 with structured Error code MAX_BATCH_SIZE_EXCEEDED carrying { limit: 100, received: N }
      - Runs the entire batch inside one outer adapter.transaction
      - For each memory item: validates required fields (title or derivable title, content), computes scope/nest/branch via the same normalizeScope + nest/branch fallback logic as the existing storeEntry, computes fingerprint, checks fingerprint against existing rows, then runs checkDuplicate for semantic match — if either dedup matches, increment `duplicates` and skip the insert; if neither matches, insert into memory_entries + memory_revisions
      - Per-row validation failures (missing content, missing derivable title) push { index, message } onto errors[] and continue processing
      - Unexpected errors (DB constraint, etc.) bubble up to the outer transaction → full rollback
      - Embedding generation happens AFTER the transaction commits (in a best-effort post-pass), to avoid stretching the transaction across slow ML calls. Failed embeddings do NOT fail the batch — they're ignored, matching existing single storeEntry tolerance
      - Returns { created, duplicates, errors, ids? } where ids is an array of length === input.length and only included when response_format === 'verbose' (default omits ids)
      - Each insert preserves the nest/branch pass-through: explicit input.nest / input.branch land verbatim on the row, fallback derivation only fires when the input is empty (matches the regression test in test/memory-nest-branch-wiring.test.js)
