Purpose: Bring the 72 LocalNest MCP tool annotations into compliance with the MCP 2025-06-18 spec so that read-only tools get correct `readOnlyHint: true`, destructive tools (delete + update_self) get `destructiveHint: true`, and idempotent-by-behavior tools (reads + dedup upserts) get `idempotentHint: true`. <constant_naming_rationale>
<!-- CONTEXT.md drift note — explains Claude's Discretion calls for transparency. -->

CONTEXT.md `<decisions>` mentioned three shared constants: `READ_ONLY_ANNOTATIONS`,
`WRITE_ANNOTATIONS`, `DELETE_ANNOTATIONS`. This plan exports FOUR constants with
two renames/additions — both changes are within "Claude's Discretion" per
CONTEXT.md `<decisions>` (last bullet: "Choose whether to lift shared annotation
constants to `tool-utils.ts` based on how many duplicates exist after the audit"):

1. **`DELETE_ANNOTATIONS` → `DESTRUCTIVE_ANNOTATIONS`** (rename)
   Reason: the destructive bucket also covers `update_self` (replaces installed
   binary — not a delete) and `memory_remove_relation` (removes a junction row —
   not a delete). Naming it after a semantic category (`destructive`) rather
   than a tool-name pattern (`delete`) keeps the constant honest when applied
   to update_self and remove_relation. The MCP spec field itself is
   `destructiveHint`, so the name mirrors the SDK terminology.

2. **`IDEMPOTENT_WRITE_ANNOTATIONS`** (new fourth bucket)
   Reason: A fourth category emerged from the audit — writes that dedup/upsert
   (returning existing row id) are additive (destructiveHint: false) AND
   idempotent (idempotentHint: true). Without this bucket, tools like
   `kg_add_triples_batch`, `kg_add_entity`, `ingest_markdown`, and
   `memory_add_relation` would need a 5th inline combination or would be
   incorrectly flagged as non-idempotent. Splitting this off creates a clean
   4-category taxonomy aligned with the MCP 2025-06-18 annotation semantics:
   (RO, WR, IW, DS) rather than the original 3-bucket (RO, WR, DELETE) model.

These renames are surfaced here so `/gsd:execute-phase` and future checker runs
don't flag them as silent drift from CONTEXT.md.
</constant_naming_rationale>
