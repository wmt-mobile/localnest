# Quick Task 260415-n69 — Summary

## Context

Dogfooding LocalNest during a token-consumption deep dive (same session) surfaced two critical problems and one optimization opportunity:

1. **10+ retrieval tools were completely broken.** Same schema regression class as the `localnest_find` bug fixed in `d682eef` — the normalizers return shapes that don't match `SEARCH_RESULT_SCHEMA`, and MCP SDK rejects every response with a structured-content validation error. Agents cannot use these tools at all.
2. **Tests only check schema identity, not runtime shape.** `test/mcp-annotations.test.js` asserts `meta.outputSchema === expectedArchetype` — reference equality. It gives false confidence: the check is "did we declare the right schema object" not "does our handler actually return that shape."
3. **Read responses leak tokens.** A full memory item is ~194 tokens. `stripEmptyFields` only gets ~5% of that. A `compact` tier would drop 50%, a `lite` tier 85%.

## What shipped

### Task 1 — Fixed 10 broken retrieval tools

New helper `ensurePaginatedShape()` in `src/mcp/common/terse-utils.ts` that overlays `total_count / count / limit / offset / has_more / next_offset / items` on top of any shape, so the PaginatedResult fields coexist with legacy field names via the schema's `.passthrough()`.

Normalizers updated (backwards-compat — legacy fields preserved):

### Task 2 — `test/mcp-runtime-shapes.test.js` (new, 16 tests)

Backfills the schema identity / runtime shape gap. For each normalizer above:

1. Constructs a realistic fixture input.
2. Calls the normalizer.
3. Parses the output against `SEARCH_RESULT_SCHEMA.data` (`zod` union of PaginatedResult and bare array) via `schema.parse()`.
4. Also asserts the legacy field is preserved for backwards compat.

Plus 3 smoke tests on `ensurePaginatedShape` itself. Any future normalizer whose output doesn't validate will fail loudly here.

### Task 3 — Expanded `stripEmptyFields`

Added to the strip lists:
- Empty strings: `agent_id`, `actor_id`, `source_ref` (previously only `nest / branch / topic / feature`).
- Null: `last_recalled_at`.
- Zero: `recall_count`.
- Empty arrays: `links`, `revisions`.

~10% additional savings per memory item on top of the Phase 27 baseline. All existing `test/terse-response.test.js` tests still pass.

### Task 4 — `response_format: 'verbose' | 'compact' | 'lite'` on 3 read tools

New `applyReadFormat()` / `applyReadFormatToItems()` helpers in `terse-utils.ts`:

- `verbose` (default): runs `stripEmptyFields` — backwards compat, no behavior change for existing callers.
- `compact`: keeps `id / title / summary / tags / kind / importance / score` plus source-specific identifiers for code (`file / start_line / end_line`) and triples (`subject / predicate / object / triple_id`). ~50% smaller per item.
- `lite`: keeps `id / title / score` plus the source identifier. ~85% smaller per item.

Nested `.memory` wrappers (recall result pattern) are projected — caller sees the identifying fields regardless of which layer they live on.

Wired into:
- `localnest_memory_list` (memory-store.ts)
- `localnest_memory_recall` (memory-workflow.ts)
- `localnest_find` (find-tools.ts)

Remaining read tools (`memory_events`, `memory_suggest_relations`, `memory_related`, `search_hybrid`, all symbol tools) keep verbose behavior — opting them into the tiers is a follow-up since `applyReadFormat` needs a per-tool decision about which nested layer to project.
