# Quick Plan 260415-n69

## Goal

Three P0 problems surfaced by dogfooding LocalNest during the token-consumption deep dive. Fix them all in one atomic commit chain:

1. **Schema regression affecting ~10 retrieval MCP tools.** The `localnest_find` schema bug fixed in `d682eef` is just one instance of a broader class — every tool that runs its result through a normalizer like `normalizeMemoryRecallResult`, `normalizeSearchHybridResult`, `normalizeSymbolResult`, etc. returns a shape that does NOT match `SEARCH_RESULT_SCHEMA` (PaginatedResult union). MCP SDK rejects every response with a structured-content validation error. Agents can't use these tools at all right now. Confirmed broken via dogfooding: `localnest_memory_recall`, `localnest_search_hybrid`, `localnest_find_definition`. Likely also broken (same pattern): `localnest_find_usages`, `localnest_find_callers`, `localnest_find_implementations`, `localnest_get_symbol`, `localnest_rename_preview`, `localnest_memory_related`, `localnest_memory_suggest_relations`, `localnest_memory_events`.

2. **Runtime shape tests don't exist.** `test/mcp-annotations.test.js` only checks schema IDENTITY (`meta.outputSchema === expectedArchetype`) — not that the handler's actual return value validates against the declared schema. This gave false confidence through the entire 0.3.0 cycle. Need a new runtime test that calls each read tool with fixture args and runs `schema.data.parse(response.data)` to prevent this regression class forever.

3. **Token waste on read responses.** Measured a full memory item at ~194 tokens. A `compact` tier (id/title/summary/tags/kind/importance/score) cuts it to ~98 tokens (50% savings). A `lite` tier (id/title) cuts it to ~29 tokens (85% savings). The existing `stripEmptyFields` only covers `nest/branch/topic/feature` — it leaves `agent_id`, `actor_id`, `source_ref`, empty `links`, `null last_recalled_at`, `recall_count:0` in the payload.

## Scope boundary

- **In scope:** correctness fixes for the ~10 broken tools, runtime validation test, expanded `stripEmptyFields` field list, new `compact` and `lite` response_format tiers on read tools, bump to `0.3.0-beta.5`, CHANGELOG.
- **Out of scope:** actually migrating every caller to use compact/lite by default (that's a docs + skill update), re-enabling full npm publish (left to a manual step), any new features beyond the three P0 items.

## Tasks

### Task 1 — Add `ensurePaginatedShape()` helper + fix all broken normalizers

**Files:** `src/mcp/common/terse-utils.ts` (new helper), `src/mcp/common/normalizers/memory.ts`, `src/mcp/common/normalizers/retrieval.ts`.

**Approach:**

- In `terse-utils.ts`, add a new exported helper:
  ```ts
  export function ensurePaginatedShape<T = unknown>(
    items: T[],
    options: { total?: number; limit?: number; offset?: number } = {}
  ): {
    total_count: number;
    count: number;
    limit: number;
    offset: number;
    has_more: boolean;
    next_offset: number | null;
    items: T[];
  } {
    const count = items.length;
    const total = Number.isFinite(options.total) ? options.total! : count;
    const limit = Number.isFinite(options.limit) ? options.limit! : count;
    const offset = Number.isFinite(options.offset) ? options.offset! : 0;
    const has_more = offset + count < total;
    return {
      total_count: total,
      count,
      limit,
      offset,
      has_more,
      next_offset: has_more ? offset + count : null,
      items
    };
  }
  ```

- Update every broken normalizer to wrap into the paginated shape AND move the non-item metadata (`query`, `symbol`, etc.) onto a side channel rather than dropping it. Two options:
  - **Option A (minimal diff):** keep the existing `{query, count, items, ...}` shape and ALSO spread `ensurePaginatedShape` fields on top. Since `SEARCH_RESULT_SCHEMA.data` uses `.passthrough()`, extra fields like `query` are tolerated. This is the lowest-risk change.
  - **Option B (meta channel):** return via `createToolResponse(ensurePaginatedShape(items), { meta: { query, symbol, ... } })` so the side channel is explicit. Higher leverage but touches more files.

  Go with **Option A** — it fixes correctness with minimal diff and keeps every existing consumer working. Field-by-field fixes:

  - `normalizeMemoryRecallResult(result, query)` → spread `ensurePaginatedShape(items, { total: result?.count, limit: items.length })` on top of the existing shape.
  - `normalizeMemoryEventsResult(result)` → same pattern.
  - `normalizeMemorySuggestionResult(result, id, threshold)` → the items are in `suggestions`, rename to `items` OR also emit `items` alongside `suggestions` (additive — safer).
  - `normalizeRelatedMemoriesResult(result, id)` → items are in `related`, same treatment.
  - `normalizeSearchHybridResult(result, query)` → items are in `results`, emit `items` alias.
  - `normalizeSymbolResult(result, symbol)` → items are split across `definitions` and `exports` — concatenate into `items` for schema, preserve both existing fields.
  - `normalizeUsageResult(result, symbol)` → items in `usages`, emit `items` alias.
  - `normalizeCallersResult` / `normalizeDefinitionResult` / `normalizeImplementationsResult` / `normalizeRenamePreviewResult` → each has its domain-specific list; concatenate or alias into `items`.

- Keep legacy field names (`results`, `definitions`, `callers`, `usages`, `suggestions`, `related`) for backwards compat with any existing callers. `.passthrough()` on the schema allows this.

**Verify:**
- `npm run check` passes.
- A new unit test in `test/search-result-shape.test.js` that directly calls each normalizer with fake input and parses the output against `SEARCH_RESULT_SCHEMA.data`.

**Done:**
- All 10+ broken tools return responses that validate against `SEARCH_RESULT_SCHEMA`.
- Legacy field names preserved for backwards compat.

### Task 4 — Add `compact` and `lite` `response_format` tiers for read tools

**Files:** `src/mcp/common/terse-utils.ts`, `src/mcp/tools/memory-store.ts`, `src/mcp/tools/memory-workflow.ts`.

**Approach:**
- In `terse-utils.ts`, add:
  ```ts
  export type ReadResponseFormat = 'verbose' | 'compact' | 'lite';

  const COMPACT_KEEP = ['id', 'title', 'summary', 'tags', 'kind', 'importance', 'score', 'raw_score'];
  const LITE_KEEP = ['id', 'title', 'score'];

  export function applyReadFormat(item: unknown, format: ReadResponseFormat = 'verbose'): unknown {
    if (format === 'verbose' || !item || typeof item !== 'object') {
      return stripEmptyFields(item);
    }
    const keep = format === 'compact' ? COMPACT_KEEP : LITE_KEEP;
    const src = item as Record<string, unknown>;
    // If item wraps a memory in .memory (recall result), project both levels
    const nested = src.memory && typeof src.memory === 'object' ? src.memory as Record<string, unknown> : null;
    const out: Record<string, unknown> = {};
    for (const key of keep) {
      if (key in src) out[key] = src[key];
      else if (nested && key in nested) out[key] = nested[key];
    }
    return out;
  }
  ```
- Wire `response_format: z.enum(['verbose', 'compact', 'lite']).default('verbose')` into the inputSchemas of `localnest_memory_list`, `localnest_memory_recall`, and `localnest_find`. Keep default `verbose` for backwards compat — callers opt-in to savings.
- Apply `applyReadFormat` to each item in the response before returning from the handler.

**Verify:**
- Calling `memory_list` with `response_format: 'compact'` returns the smaller shape; `verbose` behaves as before.
- Add a unit test in `terse-response.test.js` that asserts the byte-count reduction.

**Done:**
- Three tools expose the new tiers. Benchmark shows ~50% saving on `compact`, ~85% on `lite`.
